**Principle of the Strategy**

The principle behind this strategy is extremely simple. It can be understood as an automated market-making strategy at high frequency. After reading it, some may want to hit me because such a simple approach could generate profits. I didn’t expect it to work so well when first developing it. The key takeaway is that one should act on their ideas quickly before they lose their effectiveness.

This strategy operates based on the order book. Below is an illustration of typical order distribution in a Bitcoin exchange: 
![](https://dn-filebox.qbox.me/0d8ec18c831404d3d1c19e17299c78017abcfd48.png)

On the left side, we see buy orders showing different prices and their corresponding quantities. On the right side, sell orders are displayed similarly.

If someone wants to buy Bitcoin but does not want to place an order, they can take up an existing bid (eat the order). If the buyer has a large order, it will cause a significant number of sell orders to be filled, impacting prices. However, such impacts usually do not persist for long; others may wish to eat those sell orders and potentially revert prices in a short time. The same logic applies when selling.

For instance, if someone wants to buy 5 Bitcoins at the current price level (10377), it would push the bid up to that price. Conversely, if someone wanted to sell 5 Bitcoins, the ask price might drop to 10348. This spread is where the profit opportunity exists.

The strategy places orders slightly below the buy side and above the sell side. For example, if the current spread is from 10376.99 to 10348.01, it would result in a net profit of 28.98 based on the initial difference of 1.5.

**Further Explanations**

1. What if there's no money or coins left?
In my earlier days with limited funds, this was common; I usually placed orders one-sided but that wasn’t a big issue. Actually, you could introduce a balance logic for both assets and currencies, but the process would inevitably incur some losses as each trade is purely probabilistic. Therefore, I chose to wait on one side, but it meant wasting opportunities from the other side.
2. How was position management handled?
Initially, I used full positions for every buy/sell operation. Later, according to different parameters, positions were divided into groups; not all trades resulted in complete fills.
3. No stop-loss mechanism? 
The strategy has a comprehensive order logic but I don’t believe it needs a stop-loss (discussable), as the opportunity cost of canceling an order is too high given that every trade is a matter of probability.
4. How can this be adjusted to make coin profits?
The parameters are currently symmetrical, with 8 bid orders and 8 ask orders at a distance of one price level from each other. A slight imbalance could favor selling more coins for lower prices. If the market becomes imbalanced in your favor, you can sell at a lower price and potentially earn on the difference.

**Code Explanation**

You can find the complete code [here](https://www.fmz.com/strategy/1088). Here we will only discuss the core logic functions. This strategy has operated perfectly even in the botvs simulator despite being from over three years ago, which is truly remarkable.
The first function to be called is `GetPrice()`, which requires order depth information. Different platforms have varying depths of orders; sometimes you might not find enough volume after scanning through all levels (many 0.01 grids can cause this). The call would look like: `GetPrice('Buy')` to get the buy price.
```javascript
function GetPrice(Type,depth) {
    // _C() is the platform's error handling function
    var amountBids = 0;
    var amountAsks = 0;

    // Calculate bid price by finding the depth that meets the preset quantity
    if (Type == "Buy") {
        for (var i = 0; i < 20; i++) {
            amountBids += depth.Bids[i].Amount;
            // floatamountbuy is the preset cumulative buy order volume
            if (amountBids > floatamountbuy) {
                // Slightly above, ensuring it appears higher in queue
                return depth.Bids[i].Price + 0.01;
            }
        }
    }

    // Calculate ask price by finding the depth that meets the preset quantity
    if (Type == "Sell") {
        for (var j = 0; j < 20; j++) {
            amountAsks += depth.Asks[j].Amount;
            // floatamountsell is the preset cumulative sell order volume
            if (amountAsks > floatamountsell) {
                return depth.Asks[j].Price - 0.01;
            }
        }
    }

    // Return a default price in case no depth meets requirements to avoid errors
    return depth.Asks[0].Price;
}
```

The main function `onTick()` is as follows:
```javascript
function onTick() {
    var depth = _C(exchange.GetDepth());
    var buyPrice = GetPrice("Buy",depth);
    var sellPrice = GetPrice("Sell",depth);

    // If the spread is less than diffprice, adjust prices to ensure proper spacing
    if ((sellPrice - buyPrice) <= diffprice) {
        buyPrice -= 10;
        sellPrice += 10;
    }

    // Cancel all previous orders; new orders might have the same price, so no need to cancel them.
    CancelPendingOrders();

    // Get account information and determine available quantities
    var account = _C(exchange.GetAccount());
    var amountBuy = _N((account.Balance / buyPrice - 0.1),2);
    var amountSell = _N(account.Stocks,2);

    if (amountSell > 0.02) {
        exchange.Sell(sellPrice, amountSell);
    }
    if (amountBuy > 0.02) {
        exchange.Buy(buyPrice, amountBuy);
    }

    // Sleep to wait for the next cycle
    Sleep(sleeptime);
}
```
